# 趨勢圖篩選策略架構分析報告

本報告旨在量化比較「後端預先彙總」與「前端一次性處理」兩種資料篩選策略的成本效益，並為趨勢圖功能提供明確的架構建議。

## 分析基準與假設

- **時間範圍：** 一季 (90 天)
- **資料密度：** 平均每日 30 筆 Log 記錄
- **總資料量：** 90 天 * 30 筆/天 = 2,700 筆
- **單筆原始記錄大小：** 約 250 bytes

---

## 策略一：後端預先彙總 (Backend Pre-aggregation)

### 成本估算

*   **首次載入成本:**
    *   後端處理時間: **約 2-5 秒** (Google Apps Script 遍歷 2,700 筆記錄並執行彙總計算)。
    *   Payload 大小: **約 3.6 KB** (90 個數據點 * 約 40 bytes/點)。
*   **篩選操作成本:**
    *   後端處理時間: **約 2-5 秒** (與首次載入類似，增加一次篩選條件)。
    *   Payload 大小: **約 3.6 KB** (回傳的數據點數量不變)。

---

## 策略二：前端一次性處理 (Frontend Filtering)

### 成本估算

*   **首次載入成本:**
    *   後端處理時間: **約 1-3 秒** (僅讀取 2,700 筆原始資料並序列化)。
    *   總 Payload 大小: **約 675 KB** (2,700 筆 * 250 bytes/筆)。
    *   前端處理時間: **約 60-180 毫秒** (包含 JSON 解析與首次彙總計算)。
*   **篩選操作成本:**
    *   前端處理時間: **約 10-30 毫秒** (在記憶體中直接進行篩選與彙總，反應極快)。

---

## 比較表格

| 指標 | 策略一：後端預先彙總 | 策略二：前端一次性處理 |
| :--- | :--- | :--- |
| **首次載入 Payload 大小** | `~ 3.6 KB` | `~ 675 KB` |
| **首次載入伺服器耗時** | `~ 2-5 秒` | `~ 1-3 秒` |
| **首次載入客戶端耗時** | N/A (已由後端處理) | `~ 60-180 ms` |
| **每次篩選 Payload 大小** | `~ 3.6 KB` | 0 (純前端操作) |
| **每次篩選伺服器耗時** | `~ 2-5 秒` | 0 (純前端操作) |
| **每次篩選客戶端耗時** | N/A | `~ 10-30 ms` |
| **客戶端記憶體佔用** | 極低 | 中等 (675 KB 資料) |

---

## 最終架構建議

考量到使用者體驗 (UX) 是此應用的關鍵，**建議採用「策略二：前端一次性處理」**。

### 建議理由

1.  **使用者體驗的決定性差異：**
    *   策略一的每次篩選都需等待 `2-5 秒`，這會嚴重影響操作的流暢性，讓使用者感到明顯的卡頓。
    *   策略二在首次載入後，篩選操作近乎**即時 (`< 30 ms`)**，提供了極度流暢的互動體驗。

2.  **一次性成本 vs. 持續性成本：**
    *   策略二的 `~675 KB` 首次載入是一個**一次性成本**。雖然在網路較慢的環境下可能需要 `1-3 秒` 的初始等待，但可以使用「載入中」的提示來優化感受。
    *   策略一的伺服器延遲是**持續性成本**，每次點擊都會發生，無法避免。

3.  **可接受的資料量：**
    *   以現代的網路速度與瀏覽器性能，`~675 KB` 的資料量是在可接受的範圍內，並不會對客戶端造成過大的記憶體壓力。

### 風險與緩解

*   **風險：** 未來資料量若大幅成長 (例如，超過 10,000 筆)，首次載入時間與記憶體佔用可能會成為瓶頸。
*   **緩解措施：** 屆時可導入混合策略，例如首次載入預設顯示「最近 3 個月」的後端彙總資料，同時在背景非同步下載所有原始資料，當使用者需要篩選更長的時間範圍時，前端已有可用資料。
