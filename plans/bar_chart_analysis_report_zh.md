# Phase 1 分析報告：長條圖強化策略

本報告詳細記錄了對 `src/api_web_gui.js` 和 `src/index.html` 的程式碼審查結果，並針對為「分析控制台」長條圖添加資料標籤的兩種提議方案，提供了比較分析。

## 1. 程式碼審查摘要

### `src/api_web_gui.js` (後端邏輯)

*   **目前的靜態圖表生成 (`generateAnalysisReport`):**
    *   此函式目前透過建立一個暫時的 Google Sheet，填入匯總數據，然後使用 `Charts` 服務 (`sheet.newChart()`) 來建立長條圖和圓餅圖。
    *   產生的圖表接著在伺服器上被轉換為 Base64 編碼的 PNG 圖片 (`chart.getBlob()`)。
    *   這些圖片字串被傳送到前端進行顯示。整個渲染過程都發生在後端。長條圖的關鍵函式是 `renderNativeReport`，特別是第 628-641 行。

*   **現有的動態數據模式 (`apiGetTrendData`):**
    *   此函式為動態方法提供了一個清晰的藍圖。它在 Apps Script 內收集並處理數據，但並非生成圖片，而是回傳一個結構化的 JSON 物件，該物件專為像 Chart.js 這樣的前端圖表庫格式化。

### `src/index.html` (前端邏輯)

*   **分析控制台 (`#chartArea`):**
    *   前端從 `generateAnalysisReport` 接收 Base64 圖片字串，並使用簡單的 `<img>` 標籤（第 419-421 行）來顯示它們。前端對底層數據一無所知，只知道最終的圖片。

*   **趨勢圖 (`#trendChart`):**
    *   此部分展示了動態方法。它包含了 `Chart.js` 函式庫並使用一個 `<canvas>` 元素。
    *   `renderTrendChart` Vue 方法從 `apiGetTrendData` 接收 JSON 數據，並使用 `Chart.js` 在使用者瀏覽器中完全渲染圖表，從而實現了互動性、工具提示和動態更新。

## 2. 解決方案比較表

| 標準 | **方案一: 維持靜態圖表 (修改後端)** | **方案二: 重構為動態圖表 (前端渲染)** |
| :--- | :--- | :--- |
| **實作方式** | 修改 `api_web_gui.js` 中 `renderNativeReport` 函式內的 `setOption()` 設定，以添加資料標籤。 | 1. 建立一個新的後端函式 (`apiGetAnalysisData`)，回傳圖表數據的 JSON。<br>2. 在 `index.html` 中，用 `<canvas>` 元素取代 `<img>` 標籤。<br>3. 新增一個 Vue 方法 (`renderAnalysisChart`) 來使用 Chart.js 繪製圖表。 |
| **核心挑戰** | **不確定性。** Google Apps Script 的 `Charts` 服務客製化功能有限。**無法保證**它支援我們需要的在個別長條上顯示資料標籤的功能。如果不行，此路不通。 | **確定性。** Chart.js 透過插件 (`chartjs-plugin-datalabels`) 或內建選項，對資料標籤有強大且文檔齊全的支援。實作路徑清晰且可預測。 |
| **實作複雜度** | **低 (如果可行)。** 如果該選項存在，只需增加一兩行程式碼。 **高 (如果不可行)。** | **中。** 需要在後端和前端程式碼中進行協調更改，但專案中已存在此模式 (`apiGetTrendData` / `renderTrendChart`)。 |
| **優點** | - 程式碼更改最少。<br>- 將所有圖表生成邏輯保留在後端，與「分析控制台」目前的設計一致。<br>- PDF 匯出無需更改，因為它已經是圖片。 | - **保證成功**添加資料標籤。<br>- 為未來的互動性（懸停效果、可點擊的長條）開啟了大門。<br>- 將渲染工作轉移到客戶端，性能更佳。<br>- 更現代化且更靈活的方法。 |
| **缺點** | - 如果 Apps Script Charts 不支援該功能，**失敗風險高**。<br>- 完全犧牲任何形式的互動性或未來增強功能。<br>- 靜態圖片在高解析度顯示器上可能會顯得模糊。 | - 需要編寫/更改更多的程式碼。<br>- PDF 匯出需要重新評估。前端圖表需要先轉換為圖片，然後再傳送到後端以包含在 PDF 中。 |
| **對 PDF 報告流程影響** | **無影響。** 輸出仍然是靜態 PNG 圖片，非常適合直接插入 PDF。 | **需要額外處理。** 前端需要一個機制（例如 `canvas.toDataURL()`）將渲染的圖表轉換回圖片字串，並在觸發「下載 PDF」功能時可能需要將其傳送到後端。 |

## 3. 建議

**建議的路徑是 方案二: 重構為動態圖表 (前端渲染)。**

儘管初始實作複雜度稍高，但基於以下原因，它是更優越的選擇：

1.  **風險緩解:** 它完全消除了與 Google Apps Script `Charts` 服務限制相關的不確定性。我們確信使用 Chart.js 可以達到預期的結果。
2.  **面向未來:** 動態的、客戶端渲染的圖表是一項靈活的資產。它可以輕鬆更新以包含工具提示、鑽取功能或其他互動元素，這將在未來顯著增強「分析控制台」。
3.  **一致性:** 專案中已經有一個實作良好的動態圖表 (`trendChart`)。為分析圖表採用這種模式將使程式碼庫更加一致且易於維護。

關於 PDF 生成的挑戰已確認但可以解決。對此的標準模式是在前端設置一個「準備匯出」功能，將畫布轉換為 Base64 字串，然後可以由現有的後端 PDF 生成邏輯處理。

這種方法可靠地交付了所需的功能，同時使應用程式與現代 Web 開發實踐保持一致。
