# 快取策略效能改善預估分析 (v1)

本文件旨在量化評估通用快取策略導入後，對系統效能的預期改善效益。

---

## 1. 運算成本分析 (快取未命中 Cache Miss)

當快取未命中時，系統需要執行完整的資料讀取與處理流程。以典型的資料分析函式 `apiGetAnalysisData` 為例，其主要成本來源如下：

- **主要瓶頸：`SpreadsheetApp` I/O 操作**
    - `SpreadsheetApp.openById()`: 建立與目標試算檔案的連線。
    - `getSheetByName()`: 尋找並啟用特定的工作表。
    - `getDataRange().getValues()`: 讀取工作表中的所有資料。此操作的成本與資料量成正比，是整個流程中最耗時的部分。

- **次要成本：**
    - 資料處理與計算：在記憶體中對讀取的原始資料進行篩選、轉換與彙總。
    - `PropertiesService` / `CacheService` 寫入：將計算結果序列化後存入快取，以供後續請求使用。

**執行時間估算：**
根據歷史執行記錄與典型資料規模，一次「快取未命中」的完整操作，其執行時間預計落在 **3 至 10 秒** 之間。此時間會因當下 Google 伺服器負載與資料量而浮動。

---

## 2. 快取效益分析 (快取命中 Cache Hit)

當快取命中時，系統將繞過所有昂貴的 `SpreadsheetApp` I/O 操作，執行流程大幅簡化：

- **執行流程：**
    1.  `PropertiesService` 或 `CacheService` 讀取：直接從快取中高速讀取預先計算好的結果字串。
    2.  `JSON.parse()`: 將讀取到的 JSON 字串反序列化為 JavaScript 物件。

- **成本分析：**
    - 此流程的成本極低，僅包含記憶體與快取服務的內部操作，完全避免了外部 I/O 的延遲。

**執行時間估算：**
一次「快取命中」操作的執行時間，預計能穩定控制在 **300 毫秒以下**。

---

## 3. 量化預期改善

基於上述分析，我們可以量化快取策略帶來的效能改善。

- **延遲降低百分比：**
    - 以一個典型的 5 秒請求為例：
    - `(5s - 0.3s) / 5s ≈ 94%`
    - **預期延遲將大幅降低約 94%。**

- **首次載入成本：**
    - 「快取未命中」（通常是每日首次載入）的成本會比原本略高，因為增加了「寫入快取」的步驟。
    - 然而，這是一次性、可接受的成本。其效益將體現在所有後續的請求上，為使用者帶來顯著的體驗提升。

---

## 4. 對 Google Apps Script 配額的影響

此快取策略對 Google Apps Script 的配額消耗有著正面的影響。

- **指令碼執行時間 (Script execution time)：**
    - 快取命中將請求的執行時間從數秒縮短至數百毫秒，大幅減少了此配額的消耗。
    - 這使得系統在相同配額限制下，能夠服務更多使用者或應付更高的請求頻率，對於應付高流量使用情境至關重要。
    - 同時，這也降低了因單一使用者觸發長時間執行指令碼而影響到其他使用者操作的風險。
