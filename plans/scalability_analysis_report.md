# 可擴散性 (Scalability) 與可部署性 (Distributability) 升級分析

## 1. 系統現狀與優勢
您目前已完成配置隔離 (Config Layer) 與數據分層 (DAO Layer)，這為系統的可擴散性奠定了堅實的基礎。系統現在具備：
*   **高度可配置性：** 透過 `PropertiesService`，每個部署實例都可以獨立配置 Spreadsheet ID、Folder ID 等參數。
*   **數據操作集中：** DAO 確保了數據存取邏輯的統一。

## 2. 關鍵缺失功能與架構升級建議

為了將此單一實例 (Single Tenant) 系統轉變為可擴散的解決方案，您需要考慮以下三個關鍵領域的升級：

### 升級領域一：多租戶架構與資料隔離 (Multi-Tenancy & Isolation)

| 關鍵功能 | 說明 | 效益 |
| :--- | :--- | :--- |
| **部署腳本 (Clasp Automation)** | 目前手動部署難以擴散。應使用 `clasp` 工具或自建 GAS 函式，自動創建並部署新副本。 | 實現「一鍵部署」到不同客戶或部門，大幅降低部署成本。 |
| **租戶 ID (Tenant ID) 機制** | 每個資料來源 (Source) 應該綁定一個唯一且可配置的租戶 ID (例如部門代碼或客戶名稱)。 | 確保每個客戶使用獨立的配置和數據，避免數據混淆。 |
| **通用化 DAO** | 雖然已導入 DAO，但 `getCategoryDao(sheetNameKey)` 這類函數需要能動態決定它應該連到「哪個客戶」的配置表。 | 實現單一程式碼管理多個客戶的數據存取。 |

### 升級領域二：專業級的錯誤報告與監控 (Pro-Grade Monitoring)

| 關鍵功能 | 說明 | 效益 |
| :--- | :--- | :--- |
| **集中化日誌 (Cloud Logging)** | 應將所有 `Logger.log()` 替換為 `console.log()`，將日誌導向 Google Cloud Logging (Stackdriver)。 | 方便遠端管理和除錯，無需手動進入每個部署實例查看執行記錄。 |
| **錯誤通知機制** | 當 Webhook 或排程任務執行失敗時，自動將錯誤報告發送給管理員 (例如 Email 或 Line 通知)。 | 實現主動監控，快速反應系統故障。 |

### 升級領域三：程式碼結構與部署便利性

| 關鍵功能 | 說明 | 效益 |
| :--- | :--- | :--- |
| **前端配置抽象化** | 避免在 [`src/api_web_gui.js`](src/api_web_gui.js:1) 中寫死 `RepairCategory` 等字串。應透過 API 獲取配置，確保前端邏輯與數據配置完全解耦。 | 允許客戶客製化 Sheet 名稱而無需修改前端代碼。 |
| **工具函式集中** | 將 `formatDateValue` 等工具函式統一移至專門的 `src/utils.js` 模組，避免程式碼重複和依賴混亂。 | 保持各模組職責單一，提高代碼重用性。 |
